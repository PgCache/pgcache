# ADR-037: Generalized constraint-containment index

## Status
Accepted

## Context
ADR-024/029/030 built a per-relation index over cached queries' WHERE-clause constraints to answer one question sub-linearly: at registration, which already-cached queries' predicates *contain* a new query's predicate, so population can be skipped. That index — the subsumption index — was a single-purpose structure wired only into the registration path.

The CDC hot path asks a structurally adjacent question on every streamed row change. For each INSERT/UPDATE/DELETE on a cached relation, the writer must find which cached queries that row could affect: the in-place upsert matcher decides whether the row belongs in the shared cache table, the memo-eviction pass decides which memoized result snapshots to evict, and MV dirty-marking decides which materialized results to flip stale. Each of these was an independent full scan over *every* update query registered on the relation, re-extracting and re-checking constraints per query per row. As the per-relation query population grew (one entry per distinct literal — see the per-literal registration model), these per-operator linear scans became the dominant CDC cost, and the same scan was paid two or three times over for one row because the matcher, the memo pass, and dirty-marking did not share work. This surfaced in profiling a YCSB workload that registered ~100k entries on a single relation, where the linear per-row scan was a clear bottleneck.

The subsumption question (does query A's region contain query B's region?) and the CDC question (does this single row fall inside query A's region?) are the same containment relation evaluated at different granularities — a region probe versus a point probe. The constraint-shape partitioning that made subsumption sub-linear is exactly what is needed to narrow the CDC candidate set, but it was locked inside a registration-only type keyed to one consumer.

The hard constraint on the CDC side is asymmetric with the subsumption side: in subsumption a missed candidate only means an unnecessary population from origin (a lost optimization), whereas in CDC a missed candidate is a *stale read* — a row change that should have updated or evicted a cached result but didn't. Any shared structure therefore has to guarantee it never under-returns for the CDC consumers even where it may legitimately over-return for the subsumption consumer.

## Decision
- **Extract one reusable containment index.** The former subsumption index is generalized into `ConstraintIndex<K>` in `src/query/constraint_index.rs`, parameterized over the entry id type `K` rather than hardwired to query fingerprints, and decoupled from any single caller. It keeps the ADR-024/029/030 internals — per-table column-set classes, an equality-pure hash bucket, and a per-column complex index (`eq` / `inset` / `range_lower` / `range_upper` / `range_both` / `opaque` fallback) — but exposes them as a general-purpose containment structure.
- **Two probe modes on one structure.** `candidates(constraints)` is the region probe (a query's predicate region, enumerated over the powerset of its constraint columns) used by subsumption. `candidates_point(col_forms_fn)` is the point probe (a single row treated as an `Equal`-on-every-column degenerate query, enumerated over the existing classes) used by CDC. Both walk the same buckets and intersect per-column candidate sets; they differ only in how the probe key is generated.
- **A dedicated `eval_index` over the full CDC population.** `UpdateQueries` carries `eval_index: ConstraintIndex<Fingerprint>` alongside the registration-time `subsumption` index. It indexes the *entire* update-query population of the relation — LocalEval and PgEval queries alike — not just the subsumption-eligible single-table queries.
- **One shared probe feeds all CDC consumers.** Per row, `eval_candidates` runs a single `candidates_point` over `eval_index`, and the resulting candidate fingerprint set is threaded into the in-place upsert matcher (`update_queries_execute_batch`), the memo-eviction accumulator (`memo_frame_accumulate`), and MV removed-row dirty-marking (`mv_dirty_mark_removed_row`) — replacing each consumer's independent full-relation scan with a probe of the narrowed set.
- **Point-probe forms keep the probe non-lossy for CDC.** A row value is coerced via `row_value_forms` into every keyable interpretation (`String`, numeric `Float`, `Boolean`) and all forms are probed and unioned, so an entry keyed under either a bare numeric literal or an identity-`::text`-cast string is found. An absent, NULL, or unchanged-TOAST column yields the `Unknown` wildcard, which matches every entry constraining that column. Queries with no or partially extractable constraints land in the unconstrained class and are returned for every row.

## Rationale
- **Generalize-and-reuse over a parallel CDC structure.** A separate index purpose-built for CDC would duplicate the column-set partitioning, the complex per-column buckets, the keying/ordering logic, and their test surface — and would risk the two structures' containment semantics drifting apart. Making the existing structure generic over `K` and adding a second probe mode reuses one vocabulary of containment for both directions.
- **The point probe is the natural dual of the region probe.** A row is the limiting case of a query whose region is a single point; enumerating over the existing classes (rather than the powerset of a potentially wide row) keeps the point probe cheap while reusing the same per-column buckets the region probe already maintains.
- **Never-under-return is engineered into the point probe, not assumed.** The `Unknown` wildcard for unkeyable/absent values, the multi-form union for cross-typed literals, and the unconstrained-class fallthrough together guarantee the CDC consumers see every true candidate; the consumers' own precise checks (`update_query_matches_locally`, the PgEval evaluation, the per-query subsumption check) reject the over-returns. This is what makes it safe to share one lossy-but-complete probe across matching and eviction where a miss would be a stale read.
- **A second index instance, not a shared one, between subsumption and CDC.** Subsumption indexes only the single-table subsumption-eligible queries; CDC must see the full population including PgEval/multi-table queries. Rather than one index serving both with divergent membership rules, the reuse is at the *type* level — both are `ConstraintIndex<Fingerprint>` — while `subsumption` and `eval_index` hold different entry sets suited to their consumers.
- **Sharing the probe across the three CDC consumers** collapses what were two-to-three full scans of one row into a single probe whose result is reused, which is the actual per-row cost reduction; narrowing each scan to the candidate set is the secondary win.

## Consequences

### Positive
- CDC per-row work drops from O(queries-on-relation), paid per consumer, to O(candidates) paid once and shared across the matcher, memo eviction, and MV dirty-marking.
- One containment structure and one test surface serve both predicate subsumption and CDC matching; correctness reasoning about containment lives in one place.
- Memo eviction and MV dirtying become predicate-matched rather than relation-coarse, so unrelated cached results over the same relation are no longer evicted/dirtied by every row change.

### Negative
- A second `ConstraintIndex` per relation (`eval_index`) must be maintained on every query insert/remove in addition to `subsumption`, adding registration-time bookkeeping and memory.
- The point probe is deliberately lossy-over (wildcards for `Unknown`/unkeyable values, multi-form unions), so it over-returns candidates in those cases; the consumers' precise checks must run on the candidate set and absorb the false positives.
- The non-under-return guarantee is load-bearing and subtle: any future change to the point-probe forms, the wildcard handling, or the class enumeration that drops a true candidate becomes a stale-read bug rather than a lost optimization.
- Conservative fallbacks remain where the probe cannot reason precisely (e.g. DELETE old images that carry only the PK under REPLICA IDENTITY DEFAULT, PgEval/multi-table memos), so those paths still evict/dirty relation-coarse.

## Implementation Notes
The index lives in `src/query/constraint_index.rs` as `ConstraintIndex<K>` with `insert`/`remove`/`candidates` (region probe) and the `pub(crate) candidates_point` (point probe); `ColumnSet` keys the per-table classes and `ValueKey` is the canonical totally-ordered bucket key (numeric variants collapsed so a row value coerced to either integer or float probes the same bucket). `row_value_forms` (same module) builds a CDC row's point-probe `ColumnRange` forms from `TableMetadata`, mirroring `where_value_compare_string`'s row interpretation so the precise downstream checks agree.

`UpdateQueries` (`src/cache/update_query.rs`) holds both `subsumption` and `eval_index`. The shared CDC probe is `eval_candidates` in `src/cache/writer/cdc/invalidation.rs`; its result is consumed by `update_queries_execute_batch` and `memo_frame_accumulate` in the same module and by `mv_dirty_mark_removed_row` in `src/cache/writer/mv.rs`, with the dispatch in `src/cache/writer/cdc/dispatch.rs` computing it once per row and passing it to each. The region probe is consumed by `subsumption_check` at registration (`src/cache/writer/registration.rs`).
