# ADR-036: In-Process Response Memoization

## Status
Accepted

## Context
A cache hit on a registered query still costs work on every serve: the connection hands the request to a serve worker, the worker round-trips to the cache Postgres (or, for an MV-eligible query, scans the `q_<fingerprint>` table), and the result is re-encoded onto the wire. For the queries that dominate source-row traffic — small, stable result sets that are read far more often than their relations are written — this is pure overhead repeated on identical bytes. ADR-025 already removed most of the per-lookup cost on the cache-PG hot path and ADR-031 added the MV tier for reducing/transforming queries, but neither helps the large population of small MV-ineligible source-row queries: those are re-evaluated against the source-row cache and re-serialized on every hit, paying a serve-pool hop and a cache-DB round trip to reproduce a result that has not changed. This was not hypothetical: demo benchmarking and profiling showed the cache-DB round-trip time was a significant fraction of the work pgcache performed on a warm, high-hit-rate workload.

The opportunity is to keep the *finished, client-ready wire bytes* of such a serve in process memory and replay them straight from the connection thread on the next identical request, skipping the worker hop and the cache-DB entirely. The hard constraint is the project's no-stale-reads invariant: a memoized result is a frozen snapshot, so *any* byte-changing mutation to a relation it read must stop it being served — a stricter bar than the source-row cache's grow-only invalidation model, where an in-place UPDATE or DELETE never moves a query's generation or state yet still changes the snapshot. The added difficulty is that capture runs on a worker thread while invalidation runs on the single-threaded writer, and the in-memory invalidation signal and the cache-DB COMMIT that makes a change visible are not atomic, so a naive one-sided flag always leaves a stale window on one side of the commit.

## Decision
Add a third caching layer — an in-process result memo (`src/cache/memo.rs`) — orthogonal to the source-row cache and the MV tier, consulted *before* the source-row/MV serve decision:

- **Caches client-ready wire bytes.** A memo entry holds the contiguous `RowDescription` + `DataRow*` + `CommandComplete` core of a serve as a refcounted `Bytes`. The per-client envelope (ParseComplete / BindComplete / ReadyForQuery, and slicing off the leading RowDescription for a client that did not ask for one) is regenerated per serve around that core, so one captured core serves both simple-query and reused-prepared-statement shapes. A served hit never touches a worker or the cache DB.
- **Keyed by `(fingerprint, binary_format, shape)`** where shape is the integer LIMIT/OFFSET pair (`MemoShape`). Cross-shape slicing is unsafe without an ORDER BY, so only the exact captured shape is served; a non-integer-literal LIMIT/OFFSET is simply not memoized.
- **Capture is gated to the queries that benefit and only those.** A serve is captured only when it is a source-row serve (not an MV serve), the query is in the terminal MV-ineligible partition (`Skipped` or size-gate-failed `Ineligible` — both sticky, so a memo here never later acquires an MV, making the memo and MV tiers mutually exclusive), the fingerprint has reached `MEMO_CAPTURE_MIN_HITS` hits, and the result fits the per-entry byte cap. Capture records *every* relation the query reads via an AST walk (subquery tables included), deliberately broader than the precomputed `relation_oids`.
- **Consistency via per-slot seqlock versions.** Each dependency is a version slot (`SlotKey`). The CDC path brackets the cache-DB COMMIT in a seqlock: a pre-commit `slot_dirty_begin` takes the version even→odd ("write in progress"), and a post-commit `slot_dirty_end` takes it odd→even (a new stable version). A capture stamps the current versions of its slots *before* issuing the serve query and re-checks them at finish, refusing to stamp or store if any slot is odd or has advanced. A serve is valid iff every stamped (always-even) version still equals the current version.
- **Per-fingerprint eviction with a relation capture-window guard.** A stored entry depends only on its own `SlotKey::Memo(fingerprint)` slot, so a CDC change evicts only the memos it actually affects. The CDC frame additionally bumps a `SlotKey::Relation(oid)` slot for every touched relation; that slot is the capture-window guard — re-checked at capture finish but never stored — so a frame committing mid-capture drops the snapshot even for a fingerprint not yet in the membership map.
- **Predicate-matched eviction set.** During a CDC frame the writer accumulates which memoized fingerprints over a touched relation a row change actually affects (`memo_frame_accumulate`) and bumps `SlotKey::Memo` only for those. The precise path applies to single-table `LocalEval` memos whose WHERE can be evaluated against the CDC row; DELETEs (PK-only old image under REPLICA IDENTITY DEFAULT), `PgEval` (complex/multi-table) memos, and UPDATEs without a changed-column set fall back to evicting every memo over the relation. The bar is never-under-evict.
- **RAM-relative byte budget, lazy + swept eviction.** The total-byte budget defaults to a fraction of the detected memory budget (floored at a fixed minimum); `0` disables memoization, and the value is a dynamic setting. The store evicts stale entries lazily on lookup and via a periodic GC sweep that skips the full scan unless a slot was bumped since the last sweep.

## Rationale
- **A seqlock, not a one-sided flag.** Because the in-memory version bump and the COMMIT are not atomic, bumping only after the COMMIT serves an existing memo stale in the gap before the bump, and bumping only before lets a capture whose query ran pre-COMMIT stamp a stale version as fresh. Bracketing the COMMIT in an even/odd seqlock closes both: existing memos miss from the instant pre-commit fires (a slight, bounded over-invalidation), and captures refuse to stamp during the odd window. Every stale path collapses to a conservative miss, never a stale serve. The writer is single-threaded and processes frames serially, so begin/end are balanced per relation per frame and the version never sticks odd.
- **Per-fingerprint eviction over relation-coarse.** Coarse "any change to relation R busts every memo over R" would be correct but would evict unrelated memos on every write to a hot table. Stamping only the per-fingerprint slot and bumping it only for predicate-matched changes keeps a memo alive across writes that don't touch its result, while the cheap relation guard preserves correctness for the capture race without being stored as a serve dependency.
- **Memo competes for the same memory ceiling it consumes.** Whole-system used-memory drives the registration throttle and already counts the memo's actual resident bytes, so a RAM-relative budget lets the memo grow with available memory while remaining subordinate to the same ceiling — no separate accounting that could double-count or escape the throttle.
- **Capture only on MV-ineligible source-row serves — a marginal-utility decision.** The MV tier already collapses the expensive part of a reducing query: an aggregate that costs tens to thousands of milliseconds against the source-row cache serves from its `q_<fingerprint>` table in a fraction of that. Memoizing on top of an MV would shave perhaps another ~0.5 ms off an already-cheap serve — immaterial next to the reduction the MV already captured. The same memo budget spent on small, fast source-row queries (turning a ~1 ms serve into ~0.5 ms) is a far larger *relative* gain, and that population is much larger. Scoping memo to the MV-ineligible queries therefore maximizes aggregate serving improvement across the system rather than the peak improvement of any single query — and as a bonus avoids shadowing the MV and duplicating its storage.
- **Tuning constants are provisional.** The per-entry byte cap reflects the intended target — small, low-row-count OLTP results — rather than a measured size distribution, and is expected to become workload-driven. The budget fraction aims to use as much RAM as possible while reserving headroom for OS functionality; the fallback floor is a safe default. The capture threshold was lowered from a higher default on test evidence and may be dropped entirely, since memoizing anything served more than once is cheap.

## Consequences

### Positive
- The hottest small source-row queries serve straight from process memory on the connection thread — no serve-pool hop, no cache-DB round trip, no re-encoding — using a refcounted slice around small static envelope frames.
- Eviction is predicate-matched, so a write to a hot relation only busts the memos whose results it actually changes; unaffected memos survive.
- Strictly subordinate to memory pressure: the budget is RAM-relative and the memo's real footprint counts against the registration throttle ceiling; `0` cleanly disables it.
- Correctness degrades only toward conservative misses — every race window (mid-write slot, mid-capture commit, the disabled-toggle window) resolves to a miss, never a stale serve.

### Negative
- A third caching layer adds memory and bookkeeping (per-entry slots, per-(relation, fingerprint) membership refcounts, a seqlock bracket on every CDC frame) on top of the source-row and MV tiers.
- Only the exact captured `(format, shape)` is served; differing LIMIT/OFFSET or text-vs-binary requests for the same query each need their own capture, and non-integer-literal shapes are never memoized.
- Conservative fallbacks (DELETE, `PgEval` memos, UPDATE without a changed-column set) evict every memo over the relation, so write-heavy access to a relation underlying complex/multi-table memos will churn them.
- The memo is ephemeral and in-process: it does not survive restart and is rebuilt from cache hits, unlike the persistent MV tables.

## Implementation Notes
- **Store and capture** live in `src/cache/memo.rs`: `ResultMemo` (a `DashMap<MemoKey, MemoEntry>` of entries plus a `DashMap<SlotKey, u64>` of slot versions, a per-relation→fingerprint membership map, an atomic byte total, and a `gc_pending` flag), `MemoCapture` (in-flight worker-side capture, stamping at `begin`, fed frames as they relay, re-checking and inserting at `finish`), and `MemoKey`/`MemoShape`/`MemoEntry`/`MemoHit`/`SlotKey`.
- `SlotKey` has `Memo(fingerprint)` (the stored serve dependency) and `Relation(oid)` (the capture-window guard); `RelationMembership` and `RelationColumn` variants are reserved for future column-aware eviction and are unused.
- **Serve path**: `memo_serve_plan` / `memo_serve` in `src/cache/query_cache/serve.rs`, consulted at the top of the Ready single-request serve before the MV/source-row dispatch decision; coalesced and subsumption serves bypass the memo.
- **Capture wiring**: `memo_capture_begin` in `src/cache/serve.rs` decides eligibility and stamps read relations before the serve query is issued; the relay feeds RowDescription/DataRow/CommandComplete frames and `finish` stores on a clean serve.
- **CDC eviction**: `memo_frame_accumulate` (`src/cache/writer/cdc/invalidation.rs`) builds the per-frame predicate-matched eviction set on `WriterCore::frame_memo_evictions`; the seqlock begin/end bracket around the frame COMMIT is in `src/cache/writer/cdc/frame.rs` (relation guard slots for every touched relation, Memo slots for the accumulated set), with end always published before propagating a finalize error so a slot never sticks odd.
- **Budget**: `memo_cache_size` is a dynamic setting (`src/settings/`), defaulting via `memo_default` to a fraction of `crate::memory::total_budget_bytes()` floored at a fixed minimum; a per-entry byte cap bounds individual snapshots. `0` disables memoization.
