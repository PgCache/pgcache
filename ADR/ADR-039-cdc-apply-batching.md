# ADR-039: CDC Apply Batching and Prepared Membership-Eval

## Status
Accepted

## Context
The CDC apply path evaluates, for every INSERT/UPDATE row event, which registered queries the new row belongs to. ADR-027 split this into a Rust-side LocalEval fast path and a PgEval fallback for shapes the in-process evaluator can't decide (joins, IN lists, functions, subqueries). The PgEval branch is where the cost concentrated: each CDC row was turned into a fresh `SELECT EXISTS (<predicate>)` by inlining the row's values as literal VALUES into the resolved query, deparsing it to SQL, and issuing a `simple_query` round-trip to the cache DB — once per row, per chunk of queries. Update events additionally ran a per-row row-change SELECT (`col IS DISTINCT FROM <new>`) against the cache table to decide change-dependent invalidation.

Two costs scaled badly with write volume. First, every PgEval row paid a full parse/plan on the cache-DB backend because the inlined literals made each statement textually unique — nothing could be cached or reused across rows of the same shape. Second, the apply loop was per-row: a frame carrying K row events against one relation cost on the order of K membership round-trips plus K row-change round-trips, each a separate network flush on the writer's single eval connection. Under sustained write traffic this round-trip count is what bounds CDC apply throughput, which in turn bounds replication-slot lag — the writer is single-threaded, so slow apply directly grows the slot. This was identified by profiling the high-cardinality YCSB workload (the same write-heavy load behind the writer-bottleneck lineage of PGC-117/PGC-241), where the per-row PgEval round-trips dominated apply time.

The membership question for the PgEval shapes that dominate is per-row independent: a query without GROUP BY/HAVING, without aggregates in its projection, and referencing the changed relation only once decides each candidate row on its own. That independence is what makes both batching and statement reuse safe — the same predicate shape can be evaluated against many rows in one statement, and the statement's text no longer needs to carry the row.

## Decision
- **Prepare per-query (and per-relation) eval statements, keyed and cached.** For each batchable PgEval query, the membership predicate is rewritten so the changed table becomes a from-less subselect of `unnest()` calls over array parameters (`$1` row ordinals as `int4[]`, `$2..` one `text[]` per column in `TableMetadata.columns` order, each element cast to the column type server-side). The SQL shape is now fixed regardless of row count, so the statement is prepared once and reused across frames. Row-change detection gets the same treatment: one prepared statement per relation that joins the new tuples (carried as `unnest()` arrays) against the cache table by PK and returns the per-column `IS DISTINCT FROM` matrix. Prepared statements live in LRU caches on `WriterCdc` (`prepared_membership` keyed by `(relation_oid, fingerprint)`, `prepared_row_change` keyed by relation).
- **Batch row-change detection and membership eval per frame segment.** A frame's events are processed in segments split at Truncate boundaries; up front, one batch pass evaluates membership and one evaluates row-change for the whole segment. Rows are bucketed per relation and bound as multi-row array parameters, capped at `PG_EVAL_ROW_CHUNK` rows per statement and `PG_EVAL_CHUNK` queries per statement — collapsing `K` per-row round-trips to `⌈K/PG_EVAL_ROW_CHUNK⌉ × ⌈queries/PG_EVAL_CHUNK⌉`. Results are materialized into a `SegmentMembership` matrix that the decide pass consults per event instead of issuing its own round-trip.
- **Gate batching on per-row-independent shapes.** `pg_batchable` is classified once at registration: PgEval, single reference to the changed relation, no GROUP BY/HAVING, no aggregates in the projection. Non-batchable queries and odd-arity rows (truncated tuples) stay out of the batch and fall back to the per-row path unchanged.
- **Preserve the LocalEval / Fresh-MV partition inside the batch.** Dirtyable-MV (Fresh/Building) queries are always fully evaluated so every match dirty-marks; the membership-only ("rest") queries are evaluated only for rows that no LocalEval query and no Fresh query already matched — the batch equivalent of the per-row `if !matched` short-circuit, so locally-matched rows still cost zero PgEval round-trips.
- **Self-heal to the inline form on failure.** If a prepared statement fails (e.g. invalidated by DDL), the entry is dropped and that chunk re-runs through an inlined-VALUES `simple_query` path that produces the identical result; the statement re-prepares on next use. Schema change (`TableRegister`) clears the relation's prepared entries.

## Rationale
- **Statement preparation removes the dominant per-row parse/plan cost.** Inlined literals made every PgEval statement unique, defeating the cache-DB plan cache; moving the row into bound array parameters makes the statement text shape-stable, so steady state is all prepared-statement hits with no re-parse. This is independent of batching and would help even at one row per statement.
- **Batching attacks the round-trip count, which is the throughput bound on a single-threaded writer.** Reads run on the eval connection's pre-frame snapshot (membership) and on `db_cache` (row-change), identical to the per-row path, so collapsing many rows into one statement is snapshot-equivalent — no consistency change, only fewer flushes. The two levers compose: prepared shape-stable statements are what make a multi-row bound execution cheap to reuse.
- **Per-row independence is the safety boundary, not a heuristic.** Only shapes whose membership decides each row alone can have their surviving row ordinals projected as the per-row EXISTS answer; GROUP BY/HAVING, aggregates, and self-joins are excluded at classification, so the batch can never conflate cross-row semantics.
- **Self-healing fallback keeps correctness independent of the cache.** The prepared path is a performance layer over a behaviorally-identical inline path; any prepared-statement failure degrades to the inline form rather than to a wrong answer, so the optimization never becomes a correctness dependency.
- **An LRU bound, not unbounded caching.** Per-literal registration can produce thousands of live fingerprints; the LRU keeps the hot working set prepared, ages the rest out, and — because dropping a `Statement` closes it server-side — caps the cache-DB backend's plancache memory as a side effect. Prepare-on-first-use with an LRU is the deliberate first cut: a second-use gate (to skip preparing one-shot fingerprints) was not pursued because the LRU-thrash failure mode under high literal cardinality is owned by PGC-257 (shape-parameterized update queries), which will group a relation's update queries by literal-normalized shape and collapse the fingerprint cardinality the LRU is bounding. Gating on second use would be premature complexity ahead of that change.

## Consequences

### Positive
- PgEval CDC apply drops from O(rows) cache-DB round-trips to O(⌈rows/chunk⌉ × ⌈queries/chunk⌉), and from a per-row parse/plan to prepared-statement reuse in steady state.
- Lower CDC apply latency reduces replication-slot lag on the single-threaded writer under write-heavy traffic.
- The Fresh/rest gating means locally-matched rows still issue zero PgEval round-trips, preserving ADR-027's no-match-is-free property.
- Prepared-statement counts are bounded (LRU), and stale entries close server-side as they age out.

### Negative
- Two evaluation paths per pass (prepared + inline fallback) and a batch matrix layered over the per-row path — more apply-path code and bookkeeping to maintain.
- Cold cost: prepared-statement misses prepare sequentially before the pipelined execution; a hot working set larger than the LRU bound thrashes prepare-per-use (observable via the prepared-miss metric).
- Statement-cache memory and cache-DB plancache pressure scale with the live fingerprint working set until update queries become shape-parameterized.
- Rows of non-uniform arity (truncated tuples) and non-batchable shapes still take the per-row path, so the win is shape-dependent.

## Implementation Notes
- Batch passes and the `SegmentMembership` matrix: `src/cache/writer/cdc/segment_eval.rs` (`segment_eval`, `segment_membership_eval`, `segment_row_changes_eval`, with `membership_chunk_prepared`/`membership_chunk_inline` and `row_change_chunk_prepared`/`row_change_chunk_inline` pairs). The segment loop that drives it is in `src/cache/writer/cdc/frame.rs`; the per-row fallback eval helpers (`pg_eval_matches`, `pg_eval_any`, `cache_predicate_into`) are in `src/cache/writer/cdc/sql.rs`.
- SQL shape transforms: `src/query/transform/values.rs` — `resolved_select_node_table_replace_with_unnest` (prepared array-param form), `resolved_select_node_table_replace_with_values_batch` (inlined multi-row VALUES form), and the single-row `resolved_select_node_table_replace_with_values`. The synthetic ordinal column is `BATCH_IDX_COLUMN` (`__pgc_idx`).
- Prepared-statement caches and their key type live on `WriterCdc` in `src/cache/writer/cdc/mod.rs` (`prepared_membership: LruCache<PreparedEvalKey, Statement>`, `prepared_row_change: LruCache<Oid, Statement>`); `PreparedEvalKey` is `(relation_oid, fingerprint)` in `segment_eval.rs`.
- Constants in `src/cache/writer/cdc/mod.rs`: `PG_EVAL_ROW_CHUNK = 64` (rows per statement), `PG_EVAL_CHUNK = 32` (queries per statement), `PREPARED_EVAL_CACHE_CAPACITY = 512` (LRU bound). These are provisional first-pass sizings, expected to be tuned against the `cdc.prepared_misses` metric.
- `pg_batchable` classification: `pg_batchable_classify` in `src/cache/writer/registration.rs`, stored on `UpdateQuery.pg_batchable` (`src/cache/update_query.rs`). Membership eval runs on `cache_eval_conn`'s pre-frame snapshot; row-change eval runs on `WriterCore.db_cache`, matching the per-row connections. Metrics: `cdc.prepared_hits` / `cdc.prepared_misses` (and existing `cdc.pg_eval_hits`).
- Upstream classifier: ADR-027 (LocalEval vs PgEval) decides which queries reach this PgEval path at all; this ADR only changes how those PgEval membership and row-change checks are prepared and batched.
