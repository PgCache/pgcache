# ADR-041: Adaptive Registration Admission Control

## Status
Accepted

## Context
A cache miss for an unseen query triggers registration: the writer admits the query, populates it from origin, and maintains it live thereafter. Registration is expensive relative to a serve — it consumes in-process memory and origin SELECTs — and pgcache keys the cache on post-bind queries, so registration volume grows with parameter cardinality, not with the number of distinct query shapes. A workload that hits a wide range of literals (or a cold start across a large catalog) produces a *registration storm*: many concurrent first-misses, each demanding population, arriving faster than the single-threaded writer can drain them. Left unchecked, the storm saturates the box — the writer's command queue and the population stage both back up, replication-slot lag grows, and serve latency suffers across all queries, not just the new ones.

The pressure is fundamentally a rate-matching problem: incoming registration demand is bursty and unbounded, while the writer's sustainable registration *drain rate* is finite and varies with the workload. Crucially, the bottleneck is not fixed. Registration runs as two serialized stages on the one writer thread: command processing (draining the writer's command queue, which binds under write-heavy CDC) and population (origin SELECTs in flight, which binds when the origin is remote and round-trip-bound, a backlog the command queue stays blind to). A static admission cap cannot serve both regimes — it would be too low when the writer is idle and too high when either stage is saturated. There was already a memory-pressure throttle that refuses *new* registrations outright when the cache is over its memory budget; that gate is a binary correctness/safety stop, not a pace, and says nothing about whether the writer can keep up with the rate of admission. What was missing was throughput backpressure: pace admission to whatever the writer can actually drain right now.

## Decision
Pace new-query registration admission with a closed-loop adaptive gate that tracks the writer's measured drain rate. The gate sheds excess to origin rather than queueing it.

- **Token-bucket admission on the dispatch path.** Every first-miss that would register passes through a token bucket (`RegRateBucket` in `src/cache/reg_bucket.rs`). It consumes one token per admitted registration; a denied caller forwards the query to origin without registering (origin is the source of truth, so shedding is always safe — the query is served, just not cached this time). The bucket's burst is roughly half a second of the current rate, so brief bursts pass while the sustained rate is held to what the writer can drain. This sits *after* the memory-pressure throttle: the throttle is the hard safety stop, the bucket is the pace.
- **An adaptive refill rate, set by a BBR-lite controller.** The bucket's refill rate is not a constant; it is read fresh on every admission from shared state (`RegGate` in `src/cache/reg_gate.rs`) maintained by a periodic controller task (`reg_gate_controller` in `src/cache/runtime/reg_gate.rs`). The controller models the writer's drain rate (registrations reaching Ready per second) as the bottleneck bandwidth and the writer backlog as the queue, and AIMD-paces the admit rate toward the writer's sustainable throughput.
- **Two congestion signals, pace to the most congested.** Because the bottleneck moves between the writer's command queue and the population stage, the controller watches both: the command-queue backlog window-min and the in-flight `Loading` (populating-but-not-yet-Ready) count. It backs off multiplicatively if *either* stage's backlog stands and grows; it probes the rate up additively only when *both* are drained.
- **Probe only when shedding.** Additive increase fires only when the bucket actually denied an admission in the window — i.e. demand is bumping against the rate. A partly-warm cache offers little registration load, so small backlogs there reflect a lack of demand, not spare capacity; gating the probe on shedding stops the rate drifting up into that void.
- **No ceiling, a floor only.** There is deliberately no upper bound on the admit rate — when the writer keeps up it rises freely (effectively ungated). A minimum rate keeps registration from ever fully stalling, and the gate starts ungated (infinite rate) and at a bootstrap pace until the controller has signal.
- **Fixed-rate override.** `PGCACHE_REG_RATE` pins a constant admit rate, bypassing the controller, for testing and manual override.

## Rationale
- **Shed, don't queue.** Because origin can always serve the shed query, the back-pressure mechanism can drop excess registration demand with no correctness cost and no unbounded internal queue — the cost of shedding is one uncached serve, recoverable on the next hit once the rate recovers. A queueing design would trade the saturation for latency and memory growth in the queue itself.
- **Feedback over a static cap.** The writer's sustainable registration rate depends on the live mix of CDC write load and origin RTT, neither known ahead of time. A measured drain rate adapts to both; a static cap would be wrong in at least one regime. This is the converse of ADR-028's dynamic runtime configuration — that lets an operator change a setting at runtime; this closes a loop the system tunes itself, with no operator in it.
- **Two signals because one stage is blind to the other.** This is observed history, not a hypothetical: an earlier single-signal version (command-queue backlog only) ramped admission unbounded on a remote origin, because the population backlog is invisible to the command queue, and the in-flight `Loading` signal was added to fix it. The command queue binds under write-heavy CDC; population binds when the origin is remote and round-trip-bound — pacing to whichever stands and grows is what covers both regimes.
- **AIMD (gentle additive up, multiplicative back-off) and pacing off the backlog trend rather than the drain-rate estimate** were chosen because the single-thread drain rate is too bursty to pace at directly (it latches transient peaks above the sustainable rate); the windowed drain-rate max is retained for observability only. The specific AIMD constants (250 ms tick, bootstrap 100/s, floor 20/s, additive step 50/s, back-off 0.75, drain floor 8, loading floor 128, 10 s bandwidth window) were settled empirically during stress testing — reasonable rather than claimed optimal, and expected to be retuned as real workloads accrue.
- **Start full / start ungated.** The bucket seeds its refill clock in the past so the first admission accrues a full burst, and the gate starts at an infinite rate until the controller has signal. This addresses an under-admission seen in testing: an empty start spuriously denied legitimate cold-start registrations, forwarding them uncached when there was no actual congestion to throttle.

## Consequences

### Positive
- A registration storm is paced to what the writer can actually drain, so it can no longer saturate the box, inflate replication-slot lag, or regress serve latency for already-cached queries.
- The gate self-tunes to the live bottleneck (write-heavy CDC vs. remote-origin population) with no operator input and no static cap to misconfigure.
- Shedding is correctness-free: a denied query is served from origin, and the no-ceiling design means the rate fully reopens the moment the writer catches up, so steady-state cache coverage is unaffected.
- Reuses the existing forward-to-origin path and the existing memory-pressure throttle as a layered safety stop.

### Negative
- Under sustained overload some new queries are shed to origin and serve uncached until the rate recovers, so cold-start and high-cardinality workloads reach full cache coverage more slowly than an unbounded admit would (the explicit, accepted trade for not saturating).
- A control loop adds tuning surface (the AIMD constants and floors) and a failure mode of its own — mis-paced admission either under-admits (slow warm-up) or over-admits (the saturation it exists to prevent) if the constants are wrong for a workload.
- The controller depends on the writer faithfully publishing both backlog signals every iteration / gauge tick; a stale or drifting signal degrades the pace quality silently.
- The drain-rate "bandwidth" estimate is observability-only and not used for control, so it is dead weight if the backlog-trend control proves sufficient.

## Implementation Notes
- `RegGate` (`src/cache/reg_gate.rs`) is the shared controller/writer/dispatch state, held as `Arc<RegGate>` on `CacheStateView` (`src/cache/types.rs`). The admit rate is an f64 stored in an `AtomicU64` (`INFINITY` = ungated); `completed` (Ready count), `queue_min`/`queue_max` (command-queue backlog window), `loading` (population in-flight), and `denied` (shed count) are atomic counters/gauges.
- Signal publication: the writer folds its internal-channel depth into the window every loop iteration via `queue_observe` (`src/cache/writer/core.rs`); it bumps `completed_inc` when a registration reaches Ready (`src/cache/writer/registration.rs`); and it publishes the authoritative `Loading` count from its state scan via `loading_set` on the gauge tick (`src/cache/writer/status.rs`).
- `RegRateBucket` (`src/cache/reg_bucket.rs`) holds `(tokens, last_refill)` under a `Mutex`, reads the live rate per `try_take`, and increments `denied` on a shed. It is owned by the dispatcher (`src/cache/query_cache/`), checked at the first-miss claim point after the memory-pressure throttle.
- `reg_gate_controller` (`src/cache/runtime/reg_gate.rs`) is spawned at cache setup (`src/cache/runtime/setup.rs`) on a 250ms tick; it max-filters the drain rate over a 40-sample window for the `BtlBw` observability metric and AIMD-paces `reg_rate` from the two backlog trends gated on the shed signal. Metrics published per tick: `reg_gate_rate`, `reg_gate_btlbw`, `reg_gate_queue_min`, `reg_gate_drain_rate`, `reg_gate_loading`; the dispatch shed path increments `registration_throttled_total` (shared with the memory-pressure throttle).
